# Octopus v0.9 版本执行文档

## 状态

Release Candidate 工程里程碑已完成；候选版本为 `0.9.0rc1`。未创建正式 Tag，未执行签名/干净机和两个历史签名 RC 安装器练习。

## 目标用户

准备升级到 v1.0 的 Beta 用户、发布工程和支持人员。

## 核心问题

当前产品是否已冻结关键契约，并能自动发现阻断 v1.0 稳定性、安全、升级和支持承诺的漂移。

## 用户价值

用户获得功能冻结、可预测的兼容边界，以及只聚焦重要问题修复的正式版候选。

## 范围

- 冻结用户工作流、持久化 Schema、Local API、Search Report、诊断包和 Plugin API/权限。
- 自动审计版本、安装器默认值、冻结契约、P0/P1、支持文档和工件 SHA-256。
- 完成兼容、诊断、备份/恢复、已知问题、支持分级和紧急回滚文档。

## 非目标

- 新解析格式、新搜索模式、新 UI 主工作流或新 Plugin 能力；只有发布阻断修复可以例外。
- 伪造 Tag、签名证书、真人责任人或两个真实 RC 安装练习。

## 依赖

v0.8 的迁移回滚、诊断、兼容矩阵、长路径/权限保护和公开 Beta 工程门禁。

## 风险

签名、杀毒软件、历史安装器和企业策略仍是自动化源代码环境无法证明的发布风险。`release-audit` 会把它们保留在 `external_gates_not_executed`，防止工程通过被误读为正式外部发布通过。

## 迁移要求

v0.6–v0.8 到 RC 的持久化 Schema 不变；所有来源均必须拒绝未知较新 Schema。v1.0 前不再接受新的持久化或公开契约版本，除非它是修复 P0/P1 的唯一安全路径并同时提供迁移/回滚。

## 退出条件

功能/API/Schema/Plugin 契约冻结，自动发布审计、全仓回归、检索/性能和 unsigned 工程工件校验全部通过。正式签名 RC 练习、Tag 和真人发布职责作为未执行外部门槛留给真实发布流程。
