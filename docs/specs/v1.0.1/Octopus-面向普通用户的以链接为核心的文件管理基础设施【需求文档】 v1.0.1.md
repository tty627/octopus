# Octopus-面向普通用户的以链接为核心的文件管理基础设施【需求文档】 v1.0.1

## 一句话说清楚
一个本地优先、只读原始文件、以 Markdown 索引为核心、为人和 Agent 提供资料定位与资料组合能力的文件索引基础设施
## 定位
面向普通用户的以链接为核心的文件管理基础设施
具体点说，便是扫描用户已有文件，为文件与文件夹生成轻量索引；让人和 Agent 能快速知道“什么资料在哪里、值不值得打开、应该打开哪一页/哪一节/哪张表”；根据任务把相关文件链接组织成可读、可点击、可导出的结构
由于它以链接为核心，同时不在用户的原始文件夹中存放 AI 生成的索引摘要文件，如同八爪鱼一样，从一个管理文件中伸出各种各样的链接“触手”到各个文件夹之中，进而根据需求从目标文件夹中选取相应的文件并组合。这也是将其命名为 Octopus（八爪鱼）的原因。

## 需求
### 需求1：提高非文本型文件利用效率
用户有适合AI处理的各种文本型文件，如Markdown或是Python等文本型文件，但是在其中还有大量非文本型文件
例如：
- PDF 文件
- 手写笔记扫描件
- Markdown + 图片附件
- 封装好的GitHub 工具/CLI
- 其他难以分类的文件

这些资料的共同问题是：
- 不是文本型文件，非结构化、非标准化，在单次任务时每次调用都会消耗有限的上下文空间，且引入以后会带来上下文污染问题
- 如果是需要多次使用的非标准化文件则会带来token浪费

> 实现可复用；与主智能体的上下文实现隔离

### 需求2：保证原有的文件绝对安全性+便捷性
现有的智能体往往在其指定的工作空间中工作，但终究还会有权限上安全性的顾虑
固然可以使用诸如Claude Code的自动化等实现类似的效果，但终究需要一个目标价一个任务部署不方便
对于非信息学科相关的从业者，使用Claude Code等智能体直接执行相关任务与迁移相关任务依旧有门槛

> 从产品结构上就确保不会修改原来的文件夹，同时可以简易部署。

### 需求3：实现功能的同时减少内存开销
现在已有诸如 RAG 等专门为人工智能设计的文件管理系统。
然而，RAG 往往会在原始文件的基础之上消耗更多的内存，原文件与 RAG 的总内存可能会大于原文件大小的两倍。
这对于家用的个人PC终端而言，实现如此功能并不值得如此大的内存开销。

> 要在实现基础功能的基础之上，减少内存开销

## 核心理念：以链接为中心而不是以内容为中心
传统知识管理倾向于把内容整理成知识库，也就是把所有相关的文件都打包放在一个知识库里面，在知识库基础之上进行相应的编辑或补充。
然而，在AI时代下，我们不再有大量保存通用型知识的迫切需求。因为相关信息都可以极方便地通过智能体在互联网上查询，或是利用基础模型的能力来加以覆盖。换而言之，我们所需要储存的是独属于我们的信息，或独属于我们的信息组织形式。譬如有关我们自己的个人信息，或者是我们自己加工处理的、适合我们自己的笔记形式（如手写笔记）。
倘若这些信息本身是以文本型文件的方式加以储存的，那么现有的智能体便已足以实现相关功能。但是，倘若相关信息是以非文本型文件的形式（如图片、PDF文件或是手写内容），那便不便于现有的智能体顺路完成，而需要链接等方式来指引智能体阅读指定的文件。
更重要的是，即使1+1中的1都有，+本身这一信息的组织方式也具有重要的信息价值，譬如权重、阅读优先级、重要性排序、信息聚合关系等。
因此。“以链接为核心”或“以索引为核心”是本次任务的核心理念。
在这理念下，这个产品的重点不是复制、重写、浓缩所有信息，而是建立一个可被人和机器共同使用的索引层。

关键判断：
- 通用知识，例如高数、线代、公理、定理，模型或互联网本身已经知道。
- 用户真正需要的是：自己当时的笔记、标注、页码、上下文、文件位置。
- 因此产品所要知道的是：“相关内容在哪？”而不是“重新讲一遍全部知识”。

## 4. 存储架构
### 4.1 原始仓库 / Raw Repository
用户指定一个原始目录，例如 `D:\` 或某个学习资料目录。
特点：
- 原始文件不被修改。
- 可以包含任意格式：PDF、图片、Markdown、Word、代码、附件等。
- 只作为被扫描、被引用、被索引的来源。

### 4.2 索引仓库 / Index Repository

用户再指定一个索引仓库的地址，在这里生成索引结构。

用于保存：
- 原始文件树拓扑映射
- 每个文件夹的 AI 索引总结
- 非文本型文件的标准化Markdown文件索引
- 每次Octopus的基本settings与log【类似Obsdian的模式】
这样做的好处：
- 避免索引文件递归扫描自己，产生无限循环。
- AI 生成内容出错时不会破坏原始文件。
- 同一份 raw 数据可以生成多套不同标准的索引。
- 用户可以按不同目的组织同一批资料，例如学习索引、项目索引、Agent 工作索引。

#### 链接镜像树
这个就是索引仓库的底层架构。
功能：目录结构真实复制+每个文件不复制内容，而是在对应位置创建一个指向原文件的链接
【暂缓项：长路径、任意深度目录映射和跨平台路径极限问题暂时搁置，不作为 MVP 阻塞项。MVP 先采用常规目录深度下的链接镜像树方案；后续版本再设计稳定 ID、短路径映射或其他深路径兼容机制。】

## 5. 产品形态

### 5.1 核心CLI——这是整个系统的核心部件
核心CLI 的定位：
- Octopus MVP 是一个内置 Agentic 能力的索引器 CLI，而不是先做 UI 或纯库。
- 通过 CLI 指令，使其成为人和 Agent 都可调用的本地工具。
- 必须优先保证能调用 DeepSeek API，用于生成 Leaf、FolderNode 与任务相关的索引摘要。
- CLI 优先搭建一个可以使用的索引器：完成 Raw Repository 扫描、Index Repository 生成、索引更新、索引搜索与 Markmap 输出。
- 负责扫描、摘要、索引、筛选、生成符合 Markmap 输入规范的 Markdown，并可调用 Markmap 开源包转换为带链接的 mindmap。

### 5.2 UI界面——在核心CLI实现后，再实现UI界面
UI目标是让人可以通过图形化知识地图快速跳转到原始资料

### 5.3 支持Plugin
Plugin的作用是在现有的软件架构基础之上，继续实现额外功能。这些功能会更加面向普通用户。
譬如Timeline功能，根据目前的文件夹所有文件的时间顺序加以总结不同时间做了什么事
譬如Package功能，根据思维导图内的链接，把链接内的文件全部打包成压缩包

## 6.整体工作流程
### A.建立Repository
#### STEP 01 向用户确定Raw Repository的位置
#### STEP 02 向用户确定Index Repository的位置
#### STEP 03 扫描Raw Repository，生成文件树拓扑结构
#### STEP 04 对Raw Repository内的文件夹与文件进行AI索引总结，生成标准化Markdown文件
##### 04-1 标准化叶子索引：处理非标准化文件，生成相应索引与摘要的Markdown文件；每一个非标准化文件对应一个标准化Markdown文件
（非标准化文件，视作这个单文件索引Markdown文件的附件，存放有附件链接）
请注意：只有只有 TXT、Python、Markdown 等纯粹的文本文件才不需要生成 Markdown 叶片索引。
其他的，哪怕是 Word、Excel、PPT 等智能体也可以从中提取文本的文件，由于相关文件格式会引入其他信息，为了更好利用这些结构化的信息，依旧需要引入 Markdown 的叶子索引。
**叶子索引格式请看叶子索引格式规范性文件**

*称呼其为叶子索引的原因是，在后续生成类似思维导图的链接结构的时候，这些索引就是最小的、不可再分的叶子了。
##### 04-2 生成文件夹摘要：在所有的标准化操作都处理完以后，对每一个文件夹进行AI索引总结，生成相应的标准化Markdown文件
从下往上生成，在文件夹的索引Markdown文件中，包含了该级文件夹内所有下一级文件（夹）的摘要基本信息，同时包含了该级文件夹内所有的文件树。
【示例：
Raw Repository：
test
├─ 新建 Microsoft Excel 工作表.xlsx
├─ 新建 Microsoft Word 文档.docx
├─ 新建位图图像.bmp
├─ 新建文本文档.txt
└─ folder
   └─ 新建文本文档.txt

Index Repository：
test
├─ test文件夹的MarkDown索引总结.md
├─ 新建 Microsoft Excel 工作表.xlsx的叶子索引.md
├─ 新建 Microsoft Word 文档.docx的叶子索引.md
├─ 新建位图图像.bmp的叶子索引.md
├─ 新建文本文档.txt
└─ folder的链接
   └─ folder文件夹的MarkDown索引总结.md
   └─ 新建文本文档.txt的叶子索引.md

其中，test文件夹的MarkDown索引总结.md内容包括：
- test文件夹内所有下一级文件（夹）的摘要基本信息的进一步总结
- test文件夹内所有下一级文件（夹）的json格式抬头的提取
- test文件夹下递归到最底层的文件树拓扑结构【请注意，是Raw Repository的拓扑结构】
【具体格式类似于叶子索引格式规范文件，这里我来不及写了，等后面我再补充写，你现在可以先参考一下】

### B.使用Octopus——CLI
#### STEP 01 激活该Raw Repository的Octopus CLI
#### STEP 02 用户在CLI中输入任务指令，依据指令类型来执行不同任务；最基础与常用的指令就是 `octopus search --full "完整提示词"`，然后根据提示词来搜索相关的 Leaf 与 FolderNode 索引文件，返回相关索引链接、页码/章节/Sheet 锚点、摘要、上下文等信息
#### STEP 03 Octopus 内置 Agentic 流程根据用户指令，从 FolderNode 到 Leaf 逐级选择相关索引文件加以读取
#### STEP 04 指定格式输出
最小可用有效输出是符合 Markmap 输入规范的 Markdown。该 Markdown 必须能够通过 Markmap 开源包自动转换为带链接的 mindmap。
#### STEP 05 CLI 应调用 Markmap 开源包渲染出可以点击的 mindmap；用户在 mindmap 中点击相关链接，默认跳转到 Leaf 或 FolderNode 索引文件
请注意：Octopus 的日常检索与任务输出不直接打开非文本型原文件阅读。非文本型原文件只在生成或更新 Leaf 叶子索引时被只读解析；普通搜索、筛选、Markmap 输出和 Agentic 任务执行只读取 Leaf 与 FolderNode。

## 7.Octopus的Settings
1) 全局配置：
包括 Octopus 文件本体的所在地，以及其他适用于整个 Octopus 软件的全局设置，例如：
(a) 调用的人工智能 API；
(b) 执行相关功能的默认系统提示词。
(c) 其他适用于整个 Octopus 软件的全局设置

2) 仓库配置（局部配置）：
所在地是 index repository，也就是生成出来的镜像文件树所在的那个仓库。该配置包含：
(a) 对该仓库的个性化设置；
(b) 执行该仓库的个性化提示词；
(c) 其他针对该仓库的个性化设置信息。

## 8.Octopus的路线图
Part 1:根据此需求文档，建立起内置 Agentic 能力的索引器 CLI 的具体架构实现方式，包括但不限于选用的 CLI 框架、CLI 相关基础指令、如何调用 DeepSeek API、如何生成 Markmap 输入、如何调用 Markmap 开源包、其他实现 MVP CLI 所必需的规范
PART 2：根据 CLI 的要求，具体生成出可以运行的、实现需求文档中要求的 MVP CLI 索引器
PART 3：在 CLI 的基础之上，生成 UI 界面，UI 界面可以调用 CLI 的功能
PART 4：在UI的基础之上，生成 Plugin 功能，其中以Package与Timeline为Plugin模板
PART 5：生成安装包，研究如何部署，尤其是新手小白进行部署
