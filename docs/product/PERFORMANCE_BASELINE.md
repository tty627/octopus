# Octopus 性能基线

## 目的

本文档保存可重复的当前基线和发布门槛。所有数据必须标注环境、数据集、命令和代码版本；未标注环境的结果不能用于发布决策。

## 参考环境

- 操作系统：Windows（本地 NTFS 工作区）
- Python：3.13.7；项目支持下限为 3.12
- SQLite：当前 Python 构建的 `MAX_VARIABLE_NUMBER=32766`
- CPU：Intel Core i7-14700KF
- 内存：31.8 GiB
- 磁盘：KIOXIA EXCERIA PLUS G3 NVMe SSD；C: 可用空间 123.7 GiB
- 初始基线日期：2026-07-12；正式 v0.3 工程门槛复测：2026-07-13
- Git：`v0.3` 分支，基准时 HEAD `f2cb34a` 且工作区未提交；发布候选需在提交后复跑。

## 事务基线（优化前）

数据集为 N 个 256 字节 Markdown 写入和一个 Manifest 提交，每组使用新的临时 Index Repository。

| 文件数 | 暂存 | 提交 | 总耗时 |
| ---: | ---: | ---: | ---: |
| 25 | 97 ms | 125 ms | 240 ms |
| 50 | 178 ms | 257 ms | 472 ms |
| 100 | 379 ms | 458 ms | 895 ms |
| 200 | 733 ms | 911 ms | 1,755 ms |
| 400 | 1,502 ms | 2,047 ms | 3,629 ms |
| 800 | 3,281 ms | 6,984 ms | 10,423 ms |

已知原因：每个暂存和应用操作都曾全量重写增长中的 `record.json`。v0.3 将非持久性必需的写入合并，同时保留“触碰目标前持久化回滚意图”的安全边界。

## v0.3 优化后工程复测

同一环境和数据模型下，私有 stage 使用封闭写入，回滚意图在触碰每个目标前追加并 `fsync` 到 journal，暂存文件在同卷内原子替换目标。

| 文件数 | 暂存 | 提交 | 总耗时 |
| ---: | ---: | ---: | ---: |
| 25 | 20 ms | 33 ms | 57 ms |
| 50 | 36 ms | 56 ms | 95 ms |
| 100 | 69 ms | 101 ms | 172 ms |
| 200 | 139 ms | 192 ms | 334 ms |
| 400 | 280 ms | 1,494 ms | 1,782 ms |
| 800 | 543 ms | 1,540 ms | 2,088 ms |

800 文件单次工程复测相比优化前从 10.423 秒降至 2.088 秒，总耗时减少约 80%。

### 正式工程门槛复测（2026-07-13）

命令：`python -m benchmarks.benchmark_transactions --counts 400 800 --repeats 5 --warmups 1 --enforce`。

| 文件数 | P50 总耗时 | P95 总耗时 | P50 暂存 | P50 提交 | 峰值内存 |
| ---: | ---: | ---: | ---: | ---: | ---: |
| 400 | 2,338 ms | 2,500 ms | 591 ms | 1,791 ms | 1.75 MiB |
| 800 | 2,743 ms | 2,769 ms | 1,225 ms | 1,452 ms | 1.63 MiB |

400→800 P50 增长比为 1.173，800 文件 P95 为 2.769 秒，正式工程门槛通过。结果保存在
`.octopus-dev/benchmarks/transactions-v03-o1.json`。

增量基准命令：`python -m benchmarks.benchmark_incremental --repeats 5 --warmups 1 --enforce`。P50 为 44.7 ms，P95 为 47.0 ms，距 30 秒发布门槛有充足余量；结果保存在
`.octopus-dev/benchmarks/incremental-v03.json`。

## 搜索规模基线

在 100 个 FolderNode 候选、每个候选 500 个直接子节点时，优化前的单条 `IN (...)` 查询生成 50,000 个参数，稳定复现 `OperationalError: too many SQL variables`。
v0.3 要求子节点 ID 先去重再按不超过 1,000 个参数分批查询，50,000 个子节点不得失败。
当前自动化回归用例已构造 100×500 个子节点并通过。

## 标准数据集

- `1k-mixed`：1,000 个文件，覆盖文本、PDF、Office、图片、不支持格式和 5 层目录。
- `10k-mixed`：按 `1k-mixed` 比例扩展，包含重名、移动、删除和 pending 状态。
- `100k-metadata`：100,000 个小文件的扫描、Manifest 和搜索缓存压力集；默认不运行 OCR/AI。
- 数据集必须由生成器确定性产生，不提交个人文档或敏感内容。

`benchmarks/generate_dataset.py` 已在 2026-07-13 实际生成并核验全部三档数据集：

| 数据集 | 实际文件 | 清单条目 | 首/中/末抽样 SHA-256 | 临时模板残留 |
| --- | ---: | ---: | --- | --- |
| `1k-mixed` | 1,000 | 1,000 | 通过 | 无 |
| `10k-mixed` | 10,000 | 10,000 | 通过 | 无 |
| `100k-metadata` | 100,000 | 100,000 | 通过 | 无 |

生成物位于忽略目录 `.octopus-dev/datasets`，不会把合成数据或个人资料提交到仓库。

## 发布门槛

- 800 文件事务总耗时≤5秒，400→800 文件增长比≤2.5。
- 50,000 个子节点扩展不失败，单批 SQLite ID 参数≤1,000。
- 单文件增量更新 P95≤30秒（不含外部 AI 排队时间）。
- 已纳入发布门槛的基准相比上一通过版本退化>10%时，必须阻断或在版本文档中获得明确审批。

## 基准运行规则

每组至少预热 1 次、测量 5 次，报告 P50/P95 和峰值内存。运行期间禁止网络 AI 调用，且不将构建、依赖安装时间纳入处理耗时。
每次更新本文档时记录 Git commit/working-tree 状态，避免将不可重现的快照当作发布基线。

## v0.8 工程门槛复测（2026-07-14）

环境仍为上述 Windows 11/NTFS 参考工作站，候选版本 `0.8.0b1`，工作区为提交前 v0.8
候选。增量命令使用 1 次预热、5 次测量，P50/P95 为 36.43/37.99 ms。

事务命令使用 `--counts 400 800 --repeats 5 --warmups 1 --enforce`：

| 文件数 | P50 总耗时 | P95 总耗时 | 峰值内存 |
| ---: | ---: | ---: | ---: |
| 400 | 1,407 ms | 1,567 ms | 1.74 MiB |
| 800 | 2,922 ms | 3,013 ms | 1.63 MiB |

800 P95 较记录值 2,769 ms 退化 8.83%，仍低于 10% 相对门槛；400→800 比率 2.076
低于 2.5 绝对门槛。比率相对历史 1.173 的变化已在 v0.8 版本文档中列名批准，未静默
豁免。门禁命令为：

```powershell
python -m benchmarks.performance_gate `
  --baseline benchmarks/baselines/v0.7-release-gates.json `
  --candidate .octopus-dev/benchmarks/release-gates-v08.json `
  --approve transaction_growth_ratio --enforce
```

## v0.9 RC 工程复测（2026-07-14）

同一参考工作站、`0.9.0rc1` 提交前候选：增量 P50/P95 为 87.50/88.57 ms；400/800
文件事务 P50 为 2.254/4.392 s，P95 为 2.727/4.613 s，增长比 1.949。全部绝对门槛通过。

800 P95 和增长比相对 v0.7 记录值分别变化 66.58% 和 66.14%，门禁要求显式列名批准。
RC 没有修改事务核心，且 v0.8 同机复测为 3.013 s，说明工作站时序波动显著；本次批准只允许
工程迭代继续，不替代正式外部发布前在受控空闲机上的复跑。原始五次测量位于忽略目录
`.octopus-dev/benchmarks/transactions-v09.json`。
