# Octopus 版本、分支与兼容性规范

## 版本域

| 版本域 | 示例 | 含义 | 何时变更 |
| --- | --- | --- | --- |
| 产品版本 | `0.4.0.dev0` | CLI、包、生成器和报告的软件版本 | 每次发布 |
| Markdown Schema | `octopus_schema: 0.2` | Leaf/FolderNode 机器头兼容格式 | 机器字段兼容性变更 |
| 配置/Manifest Schema | `schema_version` | 持久化配置和状态结构 | 需要迁移时 |
| Fingerprint 版本 | `fingerprint_version` | 文件指纹算法 | 指纹语义变更 |
| 规范修订号 | `v1.0.1` | 规范文档的修订序列 | 规范文字或约束变更 |
| Prompt 版本 | 独立常量 | AI 输入/输出契约 | Prompt 语义变更 |

不同版本域不同步升级。产品从 0.3 升级到 0.4 不意味着 `octopus_schema` 必须升级。`docs/specs/v1.0.1` 中的 `v1.0.1` 不表示 Octopus 软件已发布 1.0.1。

## 产品版本规则

- 遵循 SemVer 和 PEP 440。开发中版本使用 `X.Y.Z.devN`，早期受控发布使用 `aN`/`bN`/`rcN`。
- `src/octopus/__init__.py` 是产品版本唯一代码源，Hatch 动态读取。
- CLI `version`、轮子元数据、RunReport、新生成索引的 generator/parser version 必须一致。
- 已有索引的历史 generator version 必须保留，不为了版本整齐执行破坏性重写。

## 分支模型

- `main`：只保存通过版本验收的里程碑。
- `v0.N`：当前版本集成分支；任意时刻只能有一个活动版本分支。
- `codex/v0.N/<topic>`：从当前版本分支派生的短期功能/修复分支，完成后合回当前版本分支。
- `codex/hotfix/vX.Y.Z-<topic>`：从 `main` 派生的正式版热修复，必须同时回合活动版本分支。
- 不提前创建未来版本集成分支。

## 发布流程

1. 完成版本文档的全部验收项并更新指标/性能结果。
2. 将 `X.Y.Z.devN` 切换为 `X.Y.Z`，完成 CHANGELOG 和发布文档。
3. 运行完整测试、静态检查、构建、安装烟雾测试和性能门槛。
4. 合入 `main`，创建签名 Tag `vX.Y.Z`，保存发布工件和 hash。
5. 关闭当前版本分支，从 `main` 创建下一版本分支，版本设为下一个 `.dev0`。

## 兼容性与迁移

- 0.x 期间允许发生非兼容变更，但必须提供检测、备份、dry-run、迁移和失败回滚路径。
- 迁移不得读写 Raw，只能修改 Index Repository 和全局配置。
- 未知的较新 Schema 必须拒绝写入并给出升级提示；不能将其静默降级。
- 可重建缓存（如 SQLite）不需数据迁移，但必须在源数据已确认后重建。
- v1.0 起在同一主版本内保持持久化格式向前兼容；需要破坏性迁移时升级主版本。

## 历史状态

v0.1 和 v0.2 按已完成的内部里程碑管理，不伪造发布日期或正式发布状态。v0.3 是首个
正式版本；v0.4 是当前活动开发分支。v0.4 未改变配置、Manifest、Markdown 或搜索
Schema 版本。
