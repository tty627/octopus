# Octopus 产品设计书：本地资料导航与任务装配层

> 文档状态：产品方向提案 v0.1  
> 日期：2026-07-14  
> 适用范围：Octopus 1.0 工程底座之后的产品化方向  
> 说明：本文定义目标产品形态，不代表当前版本已实现全部能力。

## 0. 执行摘要

### 0.1 最重要的产品判断

Octopus 不应被定义为另一套 RAG、GraphRAG 或“与文档聊天”软件。那些产品的主要结果是一个由模型生成的答案；Octopus 最有价值的结果应当是一个可核验、可打开、可继续工作的资料集合。

Octopus 应被定义为：

> **用户现有文件系统之上的本地资料导航与任务装配层。它不要求迁移文件，不修改原文件，通过可解释的索引帮助用户找到正确证据，并把分散资料组织成可直接用于当前任务的“任务包”。**

一句面向用户的话：

> **文件不用搬家，任务自然找到它们。**

### 0.2 产品的核心闭环

```text
找到资料 -> 判断是否有用 -> 定位到页/节/Sheet -> 组合成任务包 -> 打开、导出或交给 Agent
```

现有 Octopus 已经较好地完成了前半段的工程底座：Raw 只读、独立 Index、Leaf/FolderNode、非文本解析、本地搜索、可选 AI、Markmap、桌面端、Local API 和 Plugin。下一阶段不应继续堆叠底层能力，而应把这些能力收束成普通用户能理解的任务工作台。

### 0.3 建议的产品类别

不使用“个人知识库”“本地 RAG”或“知识图谱”作为主类别。建议使用：

- 中文类别：**本地资料导航器** / **任务资料工作台**
- 英文类别：**Local Evidence Navigator** / **File Intelligence Workspace**
- 技术副标题：面向人和 Agent 的可解释本地文件索引层

## 1. 产品诊断

### 1.1 Octopus 已经拥有的独特资产

1. **原始文件主权**：从产品结构上保证 Raw 只读，生成物进入独立 Index。
2. **不要求资料迁移**：用户可以继续使用原来的文件夹、软件和备份方式。
3. **面向异构资料**：PDF、Office、图片、扫描件、代码和普通文本可进入统一索引层。
4. **索引可读、可迁移**：Leaf 和 FolderNode 是人和机器都能读取的 Markdown，不是只能由某个向量数据库解释的黑盒。
5. **检索结果可追溯**：结果可关联文件路径、页码、标题、工作表、幻灯片及抽取质量。
6. **AI 可选**：没有 API Key 时仍然可以完成索引、更新、校验和本地搜索。
7. **可恢复的增量维护**：已有 Manifest、事务、失败重试、状态和诊断基础。
8. **可扩展动作层**：Package、Timeline、Local API 和 Plugin SDK 已经为“找到之后做什么”提供底座。

### 1.2 当前产品层的主要问题

当前桌面端更接近“仓库管理和搜索工具”，而不是用户每天愿意停留的资料工作空间：

- 首页围绕仓库、更新、校验、缓存和状态展开，用户需要先理解系统结构。
- 搜索以结果列表结束，没有自然承接“比较、筛选、组合、交付”的后续动作。
- Leaf、FolderNode、Raw、Index、Manifest 等概念对工程很重要，但不应成为普通用户的主要心智负担。
- Markmap 是输出格式，不应承担完整的知识地图体验。
- 图形化关系目前不是用户任务的一部分，容易停留在“看起来像知识图谱”的展示层。
- Timeline 和 Package 已存在，但还没有与检索主流程形成统一闭环。

因此，Octopus 当前更像一台可靠的索引引擎；下一步应把它产品化为一个“从任务出发”的工作台。

## 2. 目标用户与核心任务

### 2.1 第一目标用户

第一阶段只聚焦一类用户：

> **Windows 上拥有大量混合格式本地资料、依赖既有文件夹工作、经常需要寻找旧资料并重新组合输出的个人知识工作者。**

典型人群包括：

- 研究者、学生、教师：论文、课件、扫描笔记、数据表和代码散落在多个目录。
- 咨询、产品、项目人员：需要从历史项目中找需求、会议记录、方案和交付物。
- 设计、工程和内容从业者：资料包含图片、Office、PDF、附件和项目文件夹，不适合全部迁入笔记软件。
- 对隐私敏感的专业用户：不愿上传合同、客户资料、研究数据或个人档案。

### 2.2 用户真正雇佣 Octopus 完成的工作

用户不是为了“建立索引”而使用 Octopus，而是为了完成以下任务：

1. **我记得有这份资料，但不知道放在哪里。**
2. **我找到了几个文件，但不知道哪个最值得打开。**
3. **我知道信息在某个 PDF/Excel 里，但不想逐个翻找。**
4. **我要做一份汇报、综述或交接，需要把相关资料一次找齐。**
5. **这个项目最近新增、修改或失效了哪些资料？**
6. **我想让 Agent 使用我的资料，但不想把整个硬盘暴露给它。**

### 2.3 非目标用户

第一阶段不优先服务：

- 需要多人实时协作编辑文档的团队。
- 以写作和双链笔记为中心的重度笔记用户。
- 需要企业级实体知识图谱、权限继承和数据治理的平台客户。
- 希望 AI 直接代替自己阅读、且不关心出处和证据的纯聊天用户。
- 需要跨平台移动端同步作为第一需求的用户。

## 3. 市场差异化

### 3.1 不是功能更多，而是完成不同的工作

GraphRAG、传统 RAG、NotebookLM、Obsidian 和 Everything 都解决了相邻问题，但它们分别从“生成答案”“组织笔记”或“按名称找文件”出发。Octopus 应从“保留原始工作方式，同时让资料立即可用于任务”出发。

| 产品类型 | 核心对象 | 默认输入方式 | 主要输出 | AI 依赖 | Octopus 的差异化机会 |
| --- | --- | --- | --- | --- | --- |
| Microsoft GraphRAG | 从文本语料抽取的实体、关系、社区 | 准备语料并建立图索引 | 基于图上下文的综合答案 | 索引和查询都以 LLM 推理为核心 | 不做全局实体图；保留真实文件、目录、锚点与来源，输出可工作的资料集合 |
| 传统向量 RAG / AnythingLLM | 导入到 Workspace 的文档块 | 导入、上传或连接文档 | 与文档聊天、Agent 回答 | 通常依赖嵌入和模型 | 不以聊天为首页，不要求迁移，不把向量库作为唯一事实层，AI 关闭仍可使用 |
| NotebookLM 类来源笔记本 | 用户主动选入某个 Notebook 的来源 | 逐批添加来源 | 带来源的问答、摘要和学习内容 | 产品体验高度围绕生成能力 | 自动维护既有文件夹，覆盖长期资料，不要求用户先手工组建每个 Notebook |
| Obsidian | 用户创建和编辑的 Markdown 笔记 | 写作、双链和插件 | 笔记网络、知识库、发布 | 核心不依赖 AI | 服务“不适合改写为笔记”的异构原文件；索引而不接管用户内容 |
| Everything / 系统搜索 | 文件名、路径和文件系统元数据 | 自动索引文件系统 | 快速定位文件 | 不依赖 AI | 在名称搜索之上提供内容语义、结构锚点、质量判断和任务组合 |
| **Octopus** | **原文件、索引卡、证据锚点和任务包** | **选择已有资料目录，持续增量维护** | **可解释结果、阅读路径、任务资料包和可执行动作** | **可选且可降级** | **不搬家、不改原件、证据优先、面向任务装配** |

### 3.2 与 GraphRAG 的根本区别

GraphRAG 的强项是把文本语料抽取成实体和关系图，通过社区摘要回答跨文档、全局性的综合问题。Octopus 不应该在这个方向正面复制，而应形成四个根本差异：

1. **GraphRAG 的终点是回答，Octopus 的终点是行动。** 结果应能打开文件、定位证据、加入任务包、导出或交给工具执行。
2. **GraphRAG 的基本单位是语义实体，Octopus 的基本单位是用户拥有的资料。** 文件、文件夹、页码、Sheet、版本和路径是第一等对象。
3. **GraphRAG 生成推断关系，Octopus 优先保留可解释关系。** 文件夹归属、显式链接、引用、时间邻近、共同任务选择等关系都应说明来源。
4. **GraphRAG 为语料建立新的知识结构，Octopus 尊重并增强用户已有结构。** 原目录本身就是用户长期积累的组织信息，不应在向量化或图抽取后被抹平。

### 3.3 Octopus 应占据的心智位置

```text
文件管理器 / Everything        Octopus                RAG / GraphRAG
找到文件名和路径        ->   找证据并组装任务资料   ->   基于语料生成答案

Obsidian                   Octopus
创作和维护笔记        ->   使用已有的异构文件
```

Octopus 的机会位于“文件搜索”和“AI 问答”之间：既比文件搜索更理解内容，又比问答产品更尊重来源、路径和用户行动。

## 4. 产品原则

### 4.1 原文件主权高于便利性

- 原文件默认只读，用户已有目录仍是事实来源。
- 不要求把资料复制到专有知识库后才能使用。
- 任何复制、打包、移动或对外发送都必须是用户明确触发的动作。

### 4.2 证据先于答案

- 先告诉用户“依据是什么、在哪里、是否新鲜、抽取质量如何”，再提供模型总结。
- 所有 AI 结论必须能回到具体来源；没有来源时应明确显示为推断。
- 搜索结果不能只有相似度分数，必须显示可理解的命中原因。

### 4.3 任务先于聊天

- 首页的主输入是“查找或描述任务”，而不是一段空白聊天记录。
- 对话只用于逐步缩小任务范围，不能成为唯一交互方式。
- 每轮交互都应改变可见的证据集合、筛选条件或任务包，而不只是生成更多文本。

### 4.4 机械能力先于 AI

- 文件扫描、路径、时间、类型、结构、显式引用、变更传播和状态判断优先使用确定性方法。
- AI 只用于摘要、任务重排、关系解释和基于已选证据的生成。
- AI 不可用时，核心导航体验必须完整降级，而不是只显示错误。

### 4.5 图只在有用时出现

- 不把全库“毛线团”关系图作为首页或卖点。
- 图只展示当前任务、当前文件夹或当前时间范围内的有限关系。
- 每条边都要有类型、来源和置信度，用户可以隐藏 AI 推断边。

### 4.6 索引是基础设施，不是用户作业

- Leaf、FolderNode、Manifest、Raw、Index 等术语保留在高级设置和开发接口中。
- 普通用户看到的是资料空间、文件卡、文件夹指南、证据位置、任务包和更新状态。

## 5. 产品形态

### 5.1 最佳形态：桌面任务工作台 + 本地服务

建议的产品组合是：

- **桌面端**：普通用户的主要入口，承担搜索、证据浏览、任务包和变化视图。
- **本地后台服务**：持续索引、更新、解析和检索，不要求用户理解进程与 CLI。
- **CLI 与 Local API**：作为高级用户、Agent 和自动化的稳定接口，不承担普通用户首页。
- **人类可读 Index Repository**：作为可审计、可备份、可迁移的底层资产。
- **Plugin**：为特定工作流提供动作和视图，而不是修改核心索引语义。

### 5.2 主界面信息架构

```text
┌──────────────────────────────────────────────────────────────────────────┐
│ Octopus   [资料空间：我的研究资料]                    已同步 2 分钟前   │
├──────────────┬─────────────────────────────────────┬─────────────────────┤
│ 资料空间     │ 查找文件或描述你正在做的任务...     │ 当前证据            │
│              │ [找资料] [做任务] [看变化]           │                     │
│ 工作台       ├─────────────────────────────────────┤ 文件摘要 / 命中原因   │
│ 搜索         │ 核心资料                            │ 页码/章节/Sheet       │
│ 任务包       │  1. 需求文档.pdf  第 12 页           │ 路径与更新时间        │
│ 最近变化     │  2. 项目预算.xlsx  Sheet: 预算       │ 抽取质量与风险        │
│              │                                     │                     │
│              │ 补充资料                            │ [打开] [加入任务包]   │
│              │  3. 会议纪要.docx  “范围变更”        │ [查看相关资料]        │
├──────────────┴─────────────────────────────────────┴─────────────────────┤
│ 任务包：季度项目汇报  8 项资料  2 项待核验   [列表] [大纲] [关系] [导出] │
└──────────────────────────────────────────────────────────────────────────┘
```

设计重点：

- 左侧是用户对象和任务，不是运维功能。
- 中间是检索与任务结果，结果按作用分组，而不只是按分数排序。
- 右侧固定为证据检查器，用户无需反复打开弹窗确认来源。
- 底部或独立页面承接任务包，形成“找到之后继续工作”的闭环。
- 更新、校验、失败重试进入资料空间健康状态，异常时再提升为明显提示。

### 5.3 三种主模式

#### 找资料

用于明确目标，例如“去年客户 A 的最终报价表”。输出强调精确命中、路径、锚点、版本和快速打开。

#### 做任务

用于开放目标，例如“整理项目复盘需要的需求、关键决策、预算和最终结果”。输出不是长答案，而是一个可编辑的候选资料集合、推荐阅读顺序和缺口提示。

#### 看变化

用于持续工作，例如“这个项目最近一周发生了什么”。输出新增、修改、删除、待稳定和失败项，并能把相关变化加入任务包。

## 6. 核心用户流程

### 6.1 首次使用

1. 用户点击“添加资料空间”，选择现有文件夹。
2. Octopus 自动建议独立索引位置，普通模式不要求用户理解 Raw/Index。
3. 显示将读取的范围、默认排除项和“不会修改原文件”的明确承诺。
4. 默认使用纯本地模式；AI 增强放在完成首次有效结果之后再邀请开启。
5. 建立索引时尽早提供部分可用结果，不要求用户等待全库完成才进入主界面。
6. 完成后直接给出三个来自样本或真实目录的可点击建议查询。

首次成功标准不是“索引完成”，而是用户打开了第一份有用资料或建立了第一个任务包。

### 6.2 快速定位资料

1. 用户输入自然语言、文件名片段、人物、项目、时间或格式。
2. Octopus 返回“核心资料、相关文件夹、补充资料、需要核验”四类结果。
3. 每项结果显示：为什么命中、资料在哪里、建议打开哪里、更新时间、质量风险。
4. 用户可直接打开原文件，也可先在证据检查器中阅读索引摘要和结构。
5. 用户选择“相关资料”时，只展开有限的、带原因的关系，不跳入全库图谱。

### 6.3 从问题到任务包

示例任务：“准备新能源项目的季度汇报”。

1. 系统先把任务拆成资料槽位：目标/需求、进度、预算、风险、决策、结果。
2. 每个槽位展示候选资料和命中证据，不自动假定第一项正确。
3. 用户勾选、排除或替换资料；AI 可以建议，但不能静默决定最终集合。
4. 系统生成任务包，包含资料、锚点、推荐顺序、关系、风险和缺口。
5. 用户可选择打开全部、导出 Markdown 大纲、生成 Markmap、通过 Package 复制确认项，或把任务包交给 Agent。

### 6.4 处理资料变化

1. Octopus 在后台增量维护资料空间。
2. 常规变化不打扰用户，只更新“已同步”状态。
3. 只有影响可信度的事件才升级提醒：文件仍在编辑、索引失败、来源丢失、任务包引用过期。
4. “最近变化”按项目/文件夹/任务包聚合，而不是直接展示底层状态机事件。
5. 用户可对某个任务包执行“检查更新”，快速看到哪些证据已变化。

## 7. 产品对象模型

| 用户对象 | 含义 | 当前底层对应 | 产品上的作用 |
| --- | --- | --- | --- |
| 资料空间 Workspace | 用户选择的一组既有资料 | Raw + Index Repository | 用户管理和搜索的最高层容器 |
| 来源 Source | 用户拥有的原始文件或文件夹 | Raw 节点 | 最终可打开、可验证的事实来源 |
| 文件卡 File Card | 单个来源的紧凑索引 | Leaf | 摘要、结构、锚点、质量和来源链接 |
| 文件夹指南 Folder Guide | 某个目录的导航摘要 | FolderNode | 范围判断、下钻、阅读路径和项目语境 |
| 证据位置 Evidence Anchor | 页、节、Sheet、幻灯片、表格或文本片段 | parser evidence / citation | 让结果能定位而非只命中文件 |
| 关系 Relationship | 两项资料为何相关 | 目录、链接、引用、时间、任务共选、AI 推断 | 支撑相关资料和任务图，必须可解释 |
| 任务包 Task Packet | 围绕一个目标组织的资料集合 | 建议新增；可复用 Markmap/Package 能力 | 产品的核心差异化对象 |
| 保存视图 Saved View | 可复用的查询、筛选和排序 | 建议新增 | 如“本周新增合同”“待核验扫描件” |
| 变化事件 Change Event | 用户可理解的资料变化 | watcher / manifest / timeline | 最近变化和任务包新鲜度检查 |

任务包应成为 Octopus 在文件与答案之间新增的关键对象。它默认保存引用而不是复制内容，因此轻量、可更新、可审计，也适合交给 Agent 使用。

## 8. 关键功能设计

### 8.1 可解释检索

每条结果至少回答六件事：

1. **是什么**：文件名、类型、一句话摘要。
2. **为什么出现**：命中文件名、标题、正文证据、标签、文件夹语境或任务关系。
3. **在哪里**：资料空间、路径和可打开目标。
4. **从哪里读**：页码、章节、Sheet、幻灯片或锚点。
5. **是否可信**：更新时间、索引状态、抽取质量、OCR 或格式风险。
6. **下一步做什么**：打开、加入任务包、查看相关资料、排除或标记重点。

搜索排序可综合关键词、语义、目录语境、时间、用户行为和质量，但界面只展示可理解的排序原因，不展示无意义的内部小数分数。

### 8.2 任务包

任务包是本方案最重要的新增能力。

一个任务包包含：

- 任务名称和目标。
- 已确认资料和待核验候选。
- 每项资料的关键锚点及加入原因。
- 推荐阅读或使用顺序。
- 资料之间的关系说明。
- 缺失资料和风险提示。
- 当前引用来源的新鲜度。
- 列表、大纲、关系图和时间线四种视图。

任务包的默认动作：

- 打开所选来源。
- 导出带链接 Markdown。
- 生成可点击 Markmap。
- 调用 Package Plugin 复制用户确认的文件。
- 以受限资料清单交给本地或外部 Agent。

关键约束：任务包默认只是引用清单。只有用户执行“导出副本/打包”时才复制文件，从而避免重新制造一套难以维护的知识库。

### 8.3 关系与图

Octopus 应建设“证据关系图”，而不是追求通用知识图谱。

推荐的关系类型：

- `contains`：文件夹包含来源。
- `explicit_link`：文档或 Markdown 中存在显式链接。
- `citation`：一项资料明确引用另一项。
- `same_project`：由文件夹、配置或用户确认属于同一项目。
- `temporal_sequence`：版本或时间上前后相继。
- `selected_together`：用户曾在同一任务包中共同确认。
- `semantic_suggestion`：AI 或语义模型建议，必须带置信度并可关闭。

图的默认范围最多围绕当前任务或当前来源展开一至两层。用户应能区分事实边、行为边和推断边。

### 8.4 AI 的产品角色

建议把 AI 能力分为四级，逐级增加成本和不确定性：

| 级别 | 能力 | 是否需要 AI | 默认策略 |
| --- | --- | --- | --- |
| L0 基础导航 | 扫描、结构、元数据、关键词检索、打开目标 | 否 | 永远可用 |
| L1 索引增强 | 文件和文件夹的紧凑摘要、主题与阅读建议 | 是 | 可选、可预算 |
| L2 任务辅助 | 任务拆解、候选重排、关系说明、缺口提示 | 是 | 只消费可追溯索引 |
| L3 证据生成 | 基于已确认任务包生成报告、提纲或回答 | 是 | 必须逐条引用，默认不读取任务包外资料 |

这种分层使 Octopus 的价值不依赖某个模型供应商，也让用户能清楚知道 AI 在哪里介入、花费多少、能否关闭。

### 8.5 最近变化

最近变化不是日志查看器，而是用户工作连续性的入口：

- 按“今天、最近 7 天、最近 30 天”浏览变化。
- 按项目或文件夹聚合新增、修改和删除。
- 显示“哪些任务包受到影响”。
- 支持把变化集合直接创建为任务包，例如“本周项目变更”。
- 对仍在编辑或抽取失败的来源显示清晰的人类语言，不暴露底层状态码。

### 8.6 Plugin 的定位

Plugin 应扩展“视图和动作”，不改变 Raw 只读、索引格式和权限边界。

优先插件方向：

- Package：把任务包中的确认来源复制或压缩为交付物。
- Timeline：把来源和变化按时间组织。
- Project Handover：生成项目交接资料包与缺口清单。
- Citation Map：提取论文、报告或合同之间的显式引用。
- Duplicate/Version Review：发现重复文件和可能的版本链，但不自动删除。
- Domain Templates：为研究综述、项目复盘、合同审阅等任务提供资料槽位模板。

## 9. 交互规则

### 9.1 结果优先，不让用户等 AI

- 本地结果先出现，AI 增强完成后平滑补充，不阻塞首屏。
- 大资料空间先提供可用的局部结果，再逐步补齐。
- AI 失败时保留本地结果和已有任务包，不把整个任务标为失败。

### 9.2 高风险动作必须显式

- 打开原文件可以一步完成。
- 复制、导出、调用外部模型、发送给 Agent 或 Plugin 必须显示范围。
- 用户确认的是“本次涉及哪些来源”，而不是一次性授予整个资料空间。

### 9.3 技术状态降噪

- 正常状态只显示“已同步”和时间。
- 需要用户处理时显示“3 项资料待处理”，点击后再展开原因。
- 校验、缓存重建、迁移、锁和事务信息放入“资料空间健康”高级页面。

### 9.4 用户标记优先于 AI

- 用户可以标记“重要、过期、不要推荐、属于某项目、首选版本”。
- 这些标记必须在自动更新中稳定保留。
- 用户确认的关系与任务包选择应比 AI 推断获得更高权重。

## 10. Leaf、FolderNode 与索引仓库的产品角色

### 10.1 Leaf：文件卡的事实底座

Leaf 不应直接作为普通用户的主要阅读界面，但它应继续承担：

- 文件级摘要、类型和主题。
- 来源路径、版本和内容标识。
- 页、节、Sheet、幻灯片和表格锚点。
- 抽取质量和风险。
- 用户重点区域及可维护记录。

产品界面的“文件卡”和“证据检查器”从 Leaf 渲染，不另建一套不可审计的内容数据库。

### 10.2 FolderNode：范围导航器

FolderNode 的核心价值不是生成更长摘要，而是帮助系统和用户回答：

- 这个文件夹大致是什么项目或主题？
- 它下面有哪些类型的资料？
- 哪些子目录值得继续展开？
- 推荐按什么顺序查看？
- 哪些范围不完整、过期或仍在变化？

因此 FolderNode 应优先作为搜索路由和文件夹指南，不应在每次更新时无差别调用 AI 重写。

### 10.3 Index Repository：可携带的“资料数字影子”

Index Repository 应继续保持：

- 与 Raw 分离。
- Markdown 可读。
- 可重建，但用户标记和任务包需要备份。
- 可供 CLI、桌面端、Local API 和 Agent 共同消费。

未来可把任务包、保存视图和用户关系也以小型、可读的文件形式存入 Index，延续可迁移和不锁定的产品承诺。

## 11. 隐私、离线与 Agent 接入

### 11.1 默认隐私模型

- 默认完全本地完成扫描、索引维护和检索。
- 默认不要求账号，不上传查询、路径、内容或索引。
- 开启云端 AI 前显示将发送的索引范围、预计成本和供应商。
- 优先发送紧凑索引信号，不发送完整原文件。
- 对高度敏感的资料空间支持永久禁用外部 AI。

### 11.2 Agent 接入模型

Octopus 不应直接把整个文件系统暴露给 Agent，而应提供三层权限：

1. **搜索权限**：Agent 只能查询索引并获得脱敏候选。
2. **证据权限**：用户或策略确认后，Agent 可读取指定 Leaf/FolderNode 和锚点。
3. **来源权限**：只有任务确实需要且范围明确时，Agent 才能打开指定原文件。

任务包天然适合作为 Agent 的上下文边界：范围小、来源清楚、可复现、可撤销，也能显著减少上下文污染。

## 12. 商业与市场策略

### 12.1 滩头市场

先服务 Windows 个人专业用户，不急于进入企业知识图谱或多人协作市场。这个群体痛点明确、现有目录迁移成本高、对隐私和非文本资料有真实需求，也是当前工程能力最匹配的人群。

### 12.2 建议的产品分层

- **Local Core**：免费或开源核心；Raw 只读、本地索引、搜索、证据定位、基础任务包。
- **Pro**：高级解析、AI 任务辅助、任务模板、保存视图、版本关系、自动任务包检查。
- **Team / Self-hosted**：共享只读资料空间、权限、审计和团队任务包；不作为近期重点。

付费点应围绕“节省任务时间和专业工作流”，而不是把用户的本地索引或基础隐私能力设为付费墙。

### 12.3 对外传播

建议避免的表达：

- “用知识图谱重构你的第二大脑”
- “与所有文件聊天”
- “全能 AI 文件管家”

建议使用的表达：

- **不搬文件，也能把资料找齐。**
- **从一个问题，到一套可核验的资料。**
- **AI 可以关闭，证据永远看得见。**
- **让 Agent 只看到任务需要的文件。**

最佳演示不是让 AI 回答一个问题，而是：在一个杂乱的真实项目目录中，30 秒找到分散在 PDF、Excel、Word 和扫描件中的关键证据，组装成任务包并打开精确位置。

## 13. 明确不做什么

1. 不做完整文件管理器，不承担移动、重命名和删除用户原文件的主工作流。
2. 不做 Markdown 编辑器或 Obsidian 替代品。
3. 不默认构建全库实体知识图谱。
4. 不以聊天记录作为产品的主要信息架构。
5. 不把 AI 生成摘要当作事实来源。
6. 不在近期做多人实时协作和云端同步平台。
7. 不承诺自动理解所有二进制格式；不支持的格式应诚实提供元数据级索引和质量说明。
8. 不允许 Plugin 绕过 Raw 只读和用户确认边界。

## 14. 版本路线图

### 阶段 A：产品可信度补齐

目标：让现有引擎完全配得上“可长期维护的资料索引”。

优先事项：

- 维护日志采用追加和保留策略，用户区域永不被自动覆盖。
- 明确分离 FolderNode 的机械聚合与 AI 摘要，并对父级摘要降频。
- 为“把整个文件夹作为最小叶片”提供可发现的配置流程。
- 完成文件夹移动/重命名识别和稳定身份处理。
- 补齐递归统计、父级/索引链接、语言等当前占位字段。
- 把内部状态机与用户可见的新鲜度、失败和恢复语义对应起来。

这些不是后台洁癖，而是任务包、变化视图和长期信任成立的前提。

### 阶段 B：Octopus Navigator 1.1

目标：普通用户不理解仓库术语，也能完成“找资料并打开证据”。

交付：

- 新的工作台式桌面信息架构。
- 资料空间、文件卡、文件夹指南等用户语言。
- 固定证据检查器。
- 找资料 / 做任务 / 看变化三种入口。
- 搜索结果分组、命中原因、锚点、新鲜度和质量标识。
- 后台状态降噪和可操作异常提示。

### 阶段 C：Task Pack 1.2

目标：让 Octopus 从“搜索工具”变成“任务资料工作台”。

交付：

- 可保存、可编辑、可检查新鲜度的任务包。
- 列表和大纲视图。
- Markmap 与 Package 作为任务包导出动作。
- 基于任务模板的资料槽位和缺口提示。
- 任务包级 Agent 上下文授权。

### 阶段 D：Evidence Map 1.5

目标：在不制造图谱噪声的前提下解释资料关系和变化。

交付：

- 事实边、行为边、推断边的关系模型。
- 当前任务范围内的关系图。
- 版本链、显式引用和时间线。
- 保存视图和“受影响任务包”提醒。

### 阶段 E：Agent-ready 2.0

目标：让 Octopus 成为本地 Agent 的资料权限层和证据协议。

交付：

- 标准化任务包 API。
- 搜索、证据和来源三级授权。
- 基于任务包的可复现生成与审计。
- 可插拔本地模型和云端模型策略。
- 在个人版稳定后再评估团队自托管版本。

## 15. 优先级排序

| 优先级 | 能力 | 原因 |
| --- | --- | --- |
| P0 | 索引长期一致性与用户区域保护 | 没有信任，所有上层体验都不成立 |
| P0 | 搜索结果的证据检查器和用户化语言 | 直接改善每日核心任务 |
| P0 | 最小任务包：选择、排序、保存、导出链接 | 形成与搜索/RAG 的核心差异 |
| P1 | 最近变化和任务包新鲜度 | 提高持续使用价值 |
| P1 | 任务模板和资料槽位 | 降低开放任务的使用门槛 |
| P1 | Agent 任务包授权 | 利用现有 Local API 和索引优势 |
| P2 | 任务范围关系图 | 有价值，但不能早于任务包 |
| P2 | 更多格式和领域 Plugin | 应由真实使用数据决定 |
| 暂缓 | 全库实体知识图谱、多人协作、云同步 | 成本高且削弱当前差异化焦点 |

## 16. 成功指标

### 16.1 北极星指标

> **有效资料任务完成率**：用户发起查找或任务后，在限定时间内打开至少一个有用来源，或成功保存/导出一个包含已确认来源的任务包的比例。

### 16.2 核心指标

| 类别 | 指标 | 建议目标 |
| --- | --- | --- |
| 安全 | Raw 修改事件 | 永远为 0 |
| 激活 | 安装到首个有效来源 | 80% 用户不超过 10 分钟 |
| 定位 | 常见任务中位定位时间 | 相比文件管理器减少至少 50% |
| 质量 | Top-5 文件命中率 | 保持不低于 80%，并按真实任务扩充评测 |
| 解释 | 可说明命中原因的结果比例 | 100% |
| 证据 | AI 生成结论的来源覆盖率 | 100% 或明确标记为无来源推断 |
| 装配 | 搜索后形成任务包或打开来源的比例 | Beta 建立基线，持续提高 |
| 新鲜度 | 单文件变更到可搜索 P95 | 不超过 30 秒的既有目标 |
| 可靠性 | 非取消更新成功率 | 不低于 99% |
| 留存 | 第 4 周仍有有效搜索/任务包的资料空间 | Beta 阶段建立选择加入的匿名基线 |

指标默认在本地计算，不收集查询原文、路径或内容。真人研究和选择加入的匿名统计应与工程自动化指标明确区分。

## 17. 风险与取舍

| 风险 | 表现 | 处理原则 |
| --- | --- | --- |
| 产品变成另一个聊天壳 | 用户只能输入问题并阅读长答案 | 首页保持证据和任务包可见，生成文本只是一个视图 |
| 图谱新奇但无用 | 全库图拥挤、无法行动 | 图只围绕当前任务，限制层级并标记边来源 |
| 索引过期破坏信任 | 用户打开后发现版本不符 | 所有结果显示新鲜度，任务包可检查受影响来源 |
| AI 摘要幻觉 | 摘要替代原文件事实 | 摘要与证据分离，结论可回到锚点，风险显式显示 |
| 术语门槛 | 用户不知道 Raw、Index、Leaf | 普通界面使用资料空间、来源、文件卡和任务包 |
| 任务包变成重复文件夹 | 到处产生副本并快速过期 | 默认只保存引用，复制仅发生在显式导出 |
| 格式支持过度承诺 | 特殊格式抽取错误 | 元数据级降级、质量标记和 Plugin 扩展 |
| AI 成本不可控 | 大目录持续重写摘要 | 机械更新优先、父级降频、预算上限和调用记录 |
| 文件移动导致断链 | 路径变化后任务包失效 | 稳定内容身份、移动/重命名识别和链接修复 |
| Windows 聚焦限制增长 | 用户要求跨平台和移动端 | 先在最匹配的平台做深，Local API 为未来客户端保留接口 |

## 18. 最终建议

Octopus 最优的产品路线不是继续与 GraphRAG 比谁能构建更复杂的图，也不是与通用 RAG 比谁能生成更像人的回答。它应该把自身已有的工程优势变成一种市面上不够清晰的产品形态：

> **一个尊重原文件、长期维护资料位置与证据、并能围绕任务临时组织正确文件集合的本地工作台。**

产品主循环必须始终是：

```text
Find -> Inspect -> Assemble -> Act
查找 -> 核验 -> 组装 -> 行动
```

其中“任务包”是最值得投资的差异化对象，“证据检查器”是最需要优先改善的交互，“本地可用、AI 可选、原文件不动”是不能交换的产品承诺。
