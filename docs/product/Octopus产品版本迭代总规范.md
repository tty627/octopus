# Octopus 产品版本迭代总规范

## 文档定位

本文档是 Octopus 产品迭代的上位规范，用于统一产品目标、版本边界、优先级、验收门槛和 v1.0 路径。各版本的执行细节以 `docs/releases/`
中的独立文档为准；字段和文件格式以 `docs/specs/` 中的规范为准；已接受的技术取舍以 `docs/adr/` 为准。

## 目标用户

- 有大量 PDF、Office、图片、扫描件和代码资料，但不希望改造原有目录的个人用户。
- 需要在本地资料中定位证据、文件和阅读顺序的知识工作者。
- 需要低成本、可解释索引层的本地 Agent 和自动化工具。

## 核心用户价值

Octopus 应当让用户在不改写 Raw Repository 的前提下，更快回答三个问题：

1. 什么资料在哪里？
2. 哪些资料最值得打开，为什么？
3. 应当按什么顺序阅读或组合它们？

## 产品原则

- **Raw 绝对只读**：所有生成物、临时文件和配置只能写入 Index Repository 或用户全局配置。
- **链接优先**：优先保存位置、证据、结构和阅读路径，不复制一份新的完整知识库。
- **本地基础能力不依赖 AI**：没有 API Key 时仍可建立索引、更新、校验和本地搜索。
- **可解释与可恢复**：用户能知道处理了什么、花费多少、为什么失败、如何恢复。
- **机械处理优先**：可确定完成的扫描、聚合、链接和状态传递不消耗 AI 调用。
- **性能是发布特性**：规模、延迟和资源开销必须进入版本验收，不作为无期限后置优化。

## 优先级规则

每个版本只能有一个主用户结果。优先级依次为：数据安全 > 正确性与可恢复 > 首次成功 > 检索价值 > 性能与成本 > 界面和扩展。
必须功能、指标和验收都完成后才能退出当前版本；未达标项不能通过修改文档目标被动消除。

## 版本成熟度阶梯

| 版本 | 主题 | 主用户结果 |
| --- | --- | --- |
| v0.1 | 技术 MVP | 证明 Raw/Index 分离与 Leaf/FolderNode 路径可行 |
| v0.2 | 可恢复与可运维 | 中断或 AI 失败不破坏已承诺的索引状态 |
| v0.3 | 规模与检索可信度 | 在大批量资料上稳定运行，搜索结果可追溯到解析证据 |
| v0.4 | 安装与首次体验 | 普通 Windows 用户可在 10 分钟内获得首条有效结果 |
| v0.5 | 检索价值 Beta | 用户可更快找到目标文件并理解推荐原因 |
| v0.6 | 桌面端 Beta | 不使用命令行也能完成主要工作流 |
| v0.7 | Plugin 预览 | 第三方可在权限边界内复用索引能力 |
| v0.8 | 公开 Beta | 真实用户环境中的升级、规模和兼容性可控 |
| v0.9 | RC | 只修复阻断 1.0 发布的问题 |
| v1.0 | GA | 提供稳定、可升级、可支持的正式产品 |

## v1.0 成功标准

- Raw 修改事件为 0，并由自动化用例与发布测试持续保证。
- 支持的升级路径不要求用户删除并重建原有 Index Repository。
- 标准任务集 Top-5 文件命中率不低于 80%。
- 核心更新成功率不低于 99%，所有已定义中断点可自动恢复。
- 无 API Key 时可完成安装、索引、更新、校验和本地搜索。
- 具有明确的支持格式、性能基线、兼容政策、安装、升级和故障排查文档。

## 治理与变更

本规范变更时，必须同步检查 `ROADMAP.md`、受影响的版本文档和 `CHANGELOG.md`。改变版本目标、兼容承诺或发布门槛的决策应增加 ADR，
不得只在代码或 Issue 中隐式变更。
